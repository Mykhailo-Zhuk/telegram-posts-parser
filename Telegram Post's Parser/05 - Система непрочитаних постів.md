# 05 — Система непрочитаних постів

## Концепція

Система відстежування статусу "прочитано/непрочитано" розподілена на кілька рівнів: локальні маркери на Python (для завантаження), JSON файли та синхронізація через FastAPI.

## Рівні системи

### Рівень 1 — last_seen.json (Python, локально)

Відстежує, до якого `message_id` були завантажені пості з кожного каналу.

```json
{
  "tech_ua": 1087,
  "-1001234567890": 342,
  "startups_ukraine": 501
}
```

Логіка:
- Кожен запуск `fetcher.py` читає `last_seen.json`
- Завантажує тільки пості з `id > last_seen[channel]`
- Після завантаження оновлює маркер до максимального ID
- **Результат**: нові пості отримують `"read": false`, старі не перезавантажуються

### Рівень 2 — Поле `read` у JSON (на сервері)

Кожен пост в `posts_reviewed.json` має поле:
```json
{
  "id": 1042,
  "read": false,
  ...
}
```

**До API**: цей файл видозмінюється серверним API при позначенні постів.
**Після API**: клієнт отримує актуальний статус для всіх постів.

### Рівень 3 — React UI (браузер)

Користувач взаємодіє через:
- **Кнопка на картці** — позначити один пост як прочитаний
- **"Позначити все"** у хедері — позначить усі видимі пости
- **Фільтр "Непрочитані"** — показує тільки `read: false`
- Лічильник на фільтр-кнопці оновлюється в реальному часі

## API endpoints

### Получить все посты

```http
GET /api/posts
```

Повертає масив постів з актуальним статусом `read`:

```json
[
  {
    "id": 1042,
    "channel": "Tech UA",
    "text": "...",
    "date": "2026-04-10T09:14:00+00:00",
    "link": "...",
    "photo": "...",
    "review": "...",
    "read": false
  }
]
```

### Позначити пост як прочитаний

```http
PUT /api/posts/{post_id}/read
```

**Тіло запиту**:
```json
{ "read": true }
```

**Відповідь**:
```json
{
  "success": true,
  "data": { "id": 1042, "read": true }
}
```

Backend:
1. Читає `posts_reviewed.json`
2. Знаходить пост по ID
3. Встановлює `read = true`
4. Зберігає файл
5. Повертає оновлений пост

### Завантажити нові пості (адмін)

```http
POST /api/fetch?channel=tech_ua&limit=50
```

Запускає `fetcher.py` логіку напряму через API.

**Відповідь**:
```json
{
  "success": true,
  "data": {
    "new_posts": 12,
    "total_posts": 1042
  }
}
```

### Генерація ревю (адмін)

```http
POST /api/summarize
```

Запускає `summarize.py` для всіх постів без `review`.

**Відповідь**:
```json
{
  "success": true,
  "data": {
    "summarized": 15,
    "total": 1042
  }
}
```

## Синхронізація

```
1. Користувач завантажує сайт
   → React запитує GET /api/posts
   → Отримує актуальний JSON з сервера
   → Відображає корректний статус read

2. Користувач натискає "Позначити як прочитане"
   → React відправляє PUT /api/posts/{id}/read
   → Backend оновлює файл
   → React оновлює локальний state

3. Інший користувач на іншому пристрої
   → При завантаженні отримує актуальний статус від сервера
   → Синхронізація гарантована
```

## Стартегії для масштабування

### Варіант A — Текущая (JSON файли)

✅ Простая  
✅ Не потрібна БД  
❌ Не масштабується для мільйонів постів  
❌ Конкурентність обмежена (нові нові файли)  

Для 10k–100k постів — вполне нормально.

### Варіант B — PostgreSQL (рекомендований для масштабування)

```python
from sqlalchemy import create_engine, Column, Integer, Boolean
from sqlalchemy.orm import declarative_base

Base = declarative_base()

class Post(Base):
    __tablename__ = "posts"
    id = Column(Integer, primary_key=True)
    read = Column(Boolean, default=False)
    # ... інші поля
```

Переваги:
- Швидкі запити
- Безпечна конкурентність
- Легко додати користувачів (різні статуси для кожного користувача)

### Варіант C — Redis кеш

```python
import redis

r = redis.Redis(host="localhost", port=6379, db=0)
# Зберігати set прочитаних ID: r.sadd("read_posts", post_id)
# Проверить: r.sismember("read_posts", post_id)
```

Для реального часу + масштабування.
